# SmartDB_MCP System Description

## Overview

**SmartDB** is a universal database gateway that implements the **Model Context Protocol (MCP) server interface**. It allows MCP-compatible clients (AI agents, IDEs like Cursor, Cline) to connect and explore multiple databases with a unified interface.

### Key Features
- **Multi-Database Support**: MySQL, PostgreSQL, Oracle, SQL Server, Dameng
- **OAuth 2.0 Authentication**: Secure access with token-based authentication
- **Role-Based Permission Control**: Fine-grained access control (admin, analyst, developer, readonly)
- **Database Health Monitoring**: Connection, transaction, lock, and resource analysis
- **SQL Optimization**: Performance analysis and optimization suggestions
- **Database Migration**: Cross-database schema and data migration tools

## Available Tools (11 Total)

### 1. execute_sql
**Purpose**: Execute SQL statements (only SQL generated by sql_creator_guide)
- **Parameters**: 
  - `query` (required): SQL statement to execute
  - `pool_name` (optional): Connection pool name, default "default"
- **Features**: Permission check, audit logging, multi-statement support
- **Usage**: Only for executing validated SQL, not for direct model-generated SQL

### 2. get_db_health
**Purpose**: Analyze database health status
- **Parameters**:
  - `pool_name` (required): Connection pool name
  - `health_type` (optional): "all", "index", "connection", "blocking", "resources"
- **Features**: Returns diagnostic report with AI expert analysis prompt

### 3. get_table_desc
**Purpose**: Query table structure and index information
- **Parameters**:
  - `tables` (required): Table name(s), comma-separated
  - `pool_name` (optional): Connection pool name, default "default"
  - `database` (optional): Database name
  - `schema` (optional): Schema name
- **Features**: Returns table structure (columns, types, comments) AND index information (index name, columns, type)
- **Usage**: For viewing specific table structures and indexes, not for listing all tables

### 4. get_table_list
**Purpose**: Get accessible table list with user permissions
- **Parameters**:
  - `pool_name` (required): Connection pool name
  - `database` (required): Database name
- **Features**: Filters tables based on user permissions (allowed_tables, allowed_columns, row_filters)

### 5. get_db_version
**Purpose**: Query database version
- **Parameters**:
  - `pool_name` (required): Connection pool name

### 6. sql_creator_guide ⭐ (Recommended First Step)
**Purpose**: Generate SQL statements based on database metadata
- **Parameters**:
  - `text` (required): User's database operation requirement
  - `pool_name` (optional): Connection pool name, default "default"
  - `database` (optional): Database name
  - `schema` (optional): Schema name
- **Features**: 
  - Returns AI prompt with real metadata for SQL generation
  - **Must be called before generate any SQL**
  - Ensures syntax accuracy, field names, table names from real metadata
- **Workflow**: sql_creator_guide → generate SQL → execute_sql

### 7. sql_optimize
**Purpose**: SQL performance optimization expert
- **Parameters**:
  - `text` (required): SQL statement to optimize
  - `tables` (required): All tables involved (format: "database.table,schema.table")
  - `pool_name` (optional): Connection pool name, default "default"
- **Features**: 5-step analysis (structure, plan, logic, index, rewrite)

### 8. get_connection_list
**Purpose**: Get accessible connection pools for current user
- **Parameters**: None (uses X-Username header)
- **Features**: Returns pools based on user role permissions

### 9. get_database_list
**Purpose**: Get accessible databases for current user
- **Parameters**:
  - `pool_name` (optional): Connection pool name, default "default"
- **Features**: 
  - For admin users with "*" permission: shows actual database names from the database
  - For non-admin users: shows configured database names
  - Different SQL queries for different database types:
    - MySQL: `SHOW DATABASES`
    - PostgreSQL: `SELECT datname FROM pg_database`
    - Oracle: `SELECT DISTINCT OWNER FROM ALL_TABLES`
    - Dameng: `SELECT DISTINCT OWNER FROM ALL_TABLES ORDER BY OWNER` (fallback: `USER_TABLES` → `SYS_CONTEXT`)

### 10. get_all_database_info
**Purpose**: Get comprehensive database information
- **Parameters**: 
  - `pool_name` (optional): Connection pool name, returns all accessible pools if not specified
- **Features**: 
  - Returns database type, all accessible databases and tables for admin users
  - Returns configured databases and tables for non-admin users
  - Includes table structure, columns, comments, indexes
  - Shows row filters and column permissions

### 11. check_sql_permission
**Purpose**: Pre-check SQL execution permissions
- **Parameters**:
  - `query` (required): SQL statement to check
  - `pool_name` (optional): Connection pool name

## Authentication Methods

### 1. Direct API Key (Headers)
```json
{
  "mcpServers": {
    "smartdb": {
      "type": "streamableHttp",
      "url": "http://localhost:3000/mcp/",
      "headers": {
        "x-username": "admin",
        "x-api-key": "your_api_key_here"
      }
    }
  }
}
```

### 2. OAuth 2.0 (Web Login)
```bash
# Start server with OAuth enabled
smartdb --oauth=true

# Then access http://localhost:3000/login
# Login with configured username/password
# Use returned bearer token in headers
```

### 3. STDIO Mode (No Authentication)
```json
{
  "mcpServers": {
    "smartdb": {
      "type": "stdio",
      "command": "uv",
      "args": ["run", "-m", "core.server", "--mode", "stdio"]
    }
  }
}
```

## Role Permission System

### Role Configuration (src/config/role.json)

| Role | Access Level | SQL Operations |
|------|-------------|-----------------|
| admin | Full access to all pools, databases, tables | SELECT, INSERT, UPDATE, DELETE, CREATE, ALTER, DROP, TRUNCATE, SHOW, DESCRIBE, EXPLAIN |
| analyst | Partial read-only access | SELECT, SHOW, DESCRIBE, EXPLAIN |
| developer | Limited read/write on specific tables | SELECT, INSERT, UPDATE, DELETE, SHOW, DESCRIBE, EXPLAIN |
| readonly | Very limited access | SELECT, SHOW, DESCRIBE, EXPLAIN |

### Permission Hierarchy
```
Pool Access → Database Access → Table Access → Column Access → Row Filters
```

### Permission Control Features
- **Pool-level**: Restrict user to specific connection pools
- **Database-level**: Restrict user to specific databases within a pool
- **Table-level**: Restrict user to specific tables
- **Column-level**: Restrict user to specific columns
- **Row-level**: Filter data (e.g., `department = 'sales'`)
- **max_rows**: Limit maximum rows returned per query

## Recommended Workflow for AI Agents

### 1. Discovery Phase
```python
# Get accessible connection pools
get_connection_list()

# Get database list for a pool
get_database_list(pool_name="default")

# Get available tables
get_table_list(pool_name="default", database="mydb")
```

### 2. Schema Understanding Phase
```python
# Get database version
get_db_version(pool_name="default")

# Get table structure and indexes (combined)
get_table_desc(tables="users,orders", pool_name="default")
```

### 3. SQL Generation Phase ⭐
```python
# Get SQL generation guide with real metadata
sql_creator_guide(
    text="查询所有订单超过1000元的用户",
    pool_name="default",
    database="mydb"
)
# Returns: AI prompt with table structure, columns, types for accurate SQL generation
```

### 4. SQL Execution Phase
```python
# Execute the generated SQL
execute_sql(query="SELECT ...", pool_name="default")
```

### 5. Optimization Phase (if needed)
```python
# Optimize slow SQL
sql_optimize(
    text="SELECT * FROM orders WHERE status = 'pending'",
    tables="mydb.orders",
    pool_name="default"
)
```

### 6. Health Check Phase
```python
# Check database health
get_db_health(
    pool_name="default",
    health_type="all"
)
# Returns: AI-powered diagnostic report
```

## Error Handling

### Common Error Responses
- `错误: 缺少查询语句` - Missing required "query" parameter
- `错误: 缺少线程池名称` - Missing required "pool_name" parameter
- `❌ 权限不足: ...` - Permission denied for SQL operation
- `数据库执行错误: ...` - Database execution error

### Audit Logging
All SQL executions are logged to:
- `logs/audit/audit.log` - Full audit trail
- Includes: username, SQL, pool_name, success/failure, execution time

## Important Notes for Agents

1. **Always use sql_creator_guide first** when generating SQL - never generate SQL directly without consulting metadata
2. **Check permissions** before executing sensitive operations
3. **Use appropriate pool_name** - default is used if not specified
4. **Respect row_filters** - data is automatically filtered based on user permissions
5. **Understand role limits** - analyst/readonly cannot execute INSERT/UPDATE/DELETE
